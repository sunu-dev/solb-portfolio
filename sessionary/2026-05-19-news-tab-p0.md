## 작업 요약

### 1) 3인 전문가 병렬 회의

`Agent` 도구로 동시 호출 — 각자 다른 관점에서 동일 코드(`NewsSection.tsx`, `useStockData.ts`, `api/news/route.ts`) 분석:
- **데이터 신뢰성 / RSS / 캐싱 아키텍처 전문가** — Google News RSS 한계, 멀티 소스 fallback, circuit breaker
- **Next.js 16 / Vercel 성능 전문가** — App Router data fetching, Fluid Compute, Cache Components, 워터폴
- **프론트엔드 UX / 인지 속도 전문가** — 스켈레톤 시간 게이트, 빈 상태 vs 에러, SWR UX

세 전문가가 만장일치로 짚은 결정적 결함 4가지:

| # | 결함 | 위치 | 결과 |
|---|---|---|---|
| 1 | `cacheTimes` 버그 — `useState({})` 가 마운트마다 리셋되어 stale 판정이 항상 true | `NewsSection.tsx:75` | 탭 왕복마다 매번 fetch → 캐시 무력화 |
| 2 | 클라이언트 fetch 타임아웃 0 (서버는 8초인데 클라이언트는 무한 대기) | `useStockData.ts:fetchNewsAPI` | 회선 끊김 시 스켈레톤 30~120초 |
| 3 | CDN이 "빈 응답(200 OK + items=[])"을 `s-maxage=180`으로 5~8분 박음 | `route.ts:132-139` | 한 명 빈 응답 받으면 그 5분간 모든 사용자가 빈 화면 |
| 4 | 5-way `Promise.all` (보유 5종목 동시 fetch) — tail latency가 전체 결정 | `NewsSection.tsx:108-130` | 최악 14초 대기 |

## 결정사항

### 데이터 신뢰성 전문가 권고 중 수용/보류
- **수용 (P0)**: 빈 응답 캐시 차단 (P0-A) — 만장일치
- **보류 (P1)**: 네이버 Search API fallback — 일 25,000건 무료, 한국어 매체 SSOT. D+7에 도입 예정. 키 발급은 사용자 액션 필요
- **보류 (V1.2)**: Supabase + Cron 사전 수집 (Google 의존 제거 최종형)

### 성능 전문가 권고 중 수용/보류
- **수용 (P0)**: `cacheTimes` 버그 + `useAutoRefresh` 가드 + fan-out 묶음
- **보류 (D+7~14)**: `'use cache'` directive (Cache Components) — `next.config.ts`에 `cacheComponents: true` 활성화 필요, 라우트 정적/동적 판정 알고리즘 변경되므로 베타 후 신중 도입
- **보류 (D+14~30)**: 시장 탭을 Server Component + Suspense streaming으로 점진 분리

### UX 전문가 권고 중 수용/보류
- **수용 (P0)**: 빈/에러/네트워크 분기 + retry counter
- **보류 (D+7)**: "방금 갱신" 배지 + Pull-to-refresh + SWR optimistic 탭 전환
- **카피 결정**: 베타 D-7엔 정직형 단일 카피, P0-G에서 분기형으로 진화

### 30분 stale cap 유지
- 데이터 전문가는 10분/15분 권고했지만 우리 cron alert 신선도와의 trade-off 검토 결과 30분 유지
- "방금 갱신" 배지(P1)와 결합되면 사용자 인지된 신선도는 30분 cap이어도 충분
