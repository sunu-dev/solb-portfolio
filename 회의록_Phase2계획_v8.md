## 2부: 분석 알고리즘, 정말 필요한가?

### 주식 고수

> **현재 규칙 기반 분석의 한계:**
>
> | 현재 (규칙 기반) | 진짜 AI가 하면 |
> |-----------------|---------------|
> | "RSI 32, 과매도" → 항상 같은 말 | RSI 32이지만 실적 발표 앞두고 있어서 추가 하락 가능성도 고려 |
> | "골든크로스 발생" → 기계적 감지 | 거래량이 동반되지 않은 골든크로스는 신뢰도 낮다는 맥락 |
> | "볼린저 하단 근접" → 단순 위치 | 밴드 수축 중인지 확장 중인지에 따라 의미가 다름 |
> | 뉴스와 연결 안 됨 | "실적 발표 D-7인데 RSI 과매도 → 실적 기대감으로 반등 가능성" |
>
> **결론: 알고리즘은 "있으면 좋고, 없으면 아쉬운" 수준.** 초보자에게는 현재 수준도 충분히 유용합니다.

### 애널리스트

> **분석 알고리즘이 필요한 시점:**
>
> 1. **사용자가 "이 분석 맨날 똑같아요"라고 피드백할 때** → Claude API 연동 필요
> 2. **종목이 20개 이상으로 늘어날 때** → 개별 종목마다 맞춤 분석 필요
> 3. **유료 구독을 시작할 때** → 무료/유료 차별점으로 AI 분석
>
> **지금 당장 필요하지 않은 이유:**
> - 사용자가 아직 적음 (피드백 없음)
> - 규칙 기반도 초보자에겐 충분히 새로운 정보
> - Claude API 비용이 사용자당 월 ~$1~3 발생

### PM

> **분석 알고리즘 로드맵:**
>
> | 단계 | 시기 | 방식 | 비용 |
> |------|------|------|------|
> | Level 1 (현재) | 지금 | 규칙 기반 20가지 | $0 |
> | Level 1.5 | 1~2주 후 | 규칙 조합 고도화 (복합 조건) | $0 |
> | Level 2 | 유저 50명+ | Claude Haiku API (일일 리포트) | ~$10/월 |
> | Level 3 | 유료 출시 시 | Claude Sonnet (심층 분석 + 뉴스 연계) | ~$30/월 |

## 3부: Phase 2에서 진짜 해야 할 것

### 전원 토론

> **질문: Phase 2의 목표는 무엇인가?**
>
> **A안: 기능 강화** (AI 분석 알고리즘, 차트 고도화)
> **B안: 사용자 확보** (로그인, 개인화, 공유 기능)
> **C안: 수익화 준비** (프리미엄 구독, 결제)
>
> **합의: B안 우선 — 사용자 확보가 먼저.**
>
> 이유:
> - 기능이 아무리 좋아도 사용자가 없으면 의미 없음
> - 현재 기능만으로 MVP는 충분
> - 로그인이 있어야 사용자 수를 측정할 수 있음
> - 로그인이 있어야 유료 전환이 가능

### UX 전문가

> **Phase 2 핵심: "다시 돌아오게 만들기"**
>
> 지금은 URL 아는 사람만 사용. Phase 2에서는:
> 1. **로그인** → 내 데이터를 안전하게 저장
> 2. **온보딩** → 처음 온 사용자가 3분 안에 가치를 느끼게
> 3. **알림** → 앱 안 열어도 중요한 변화를 알려줌
> 4. **공유** → "이 앱 써봐" 할 수 있는 기능

## 5부: Level 1.5 분석 알고리즘 — 지금 바로 할 수 있는 것

### 애널리스트 + 주식 고수

> 현재 알림 엔진의 각 조건은 **단독**으로만 체크됨.
> Level 1.5는 **복합 조건**을 분석하여 더 의미 있는 인사이트를 생성:
>
> | # | 복합 조건 | 의미 | 신뢰도 |
> |---|----------|------|--------|
> | 1 | RSI < 30 + 볼린저 하단 + 거래량 증가 | "여러 지표가 동시에 반등 신호" | 높음 |
> | 2 | 골든크로스 + MACD 상향 + 거래량 증가 | "강한 상승 전환 신호" | 높음 |
> | 3 | RSI > 70 + 볼린저 상단 + 거래량 감소 | "과열 + 관심 감소 → 조정 가능성" | 높음 |
> | 4 | 데드크로스 + MACD 하향 + 거래량 증가 | "강한 하락 전환 신호" | 높음 |
> | 5 | 평단가 이탈 + RSI < 30 | "많이 빠졌지만 반등 가능성 있는 구간" | 중간 |
> | 6 | 52주 신저가 + 거래량 급증 | "패닉셀 or 세력 매집 → 주의깊게 관찰" | 중간 |
> | 7 | 목표가 근접 + RSI > 60 | "목표 근접이고 아직 과열 아님 → 좀 더 기다려도" | 중간 |
> | 8 | 횡보 + 볼린저 밴드 수축 | "곧 큰 움직임이 올 수 있음" | 중간 |
>
> **이건 Claude API 없이도 가능합니다.** 기존 함수를 조합하면 됩니다.

## 6부: 기술 아키텍처 — Supabase 도입

### 백엔드 전문가

> **Supabase 도입 시 변경점:**
>
> ```
> 현재:
> [Next.js] → localStorage
>          → Finnhub API (직접)
>          → Yahoo Finance (/api/kr-quote)
>
> Phase 2:
> [Next.js] → Supabase Auth (로그인)
>          → Supabase DB (포트폴리오 저장)
>          → Supabase Edge Functions (API 프록시 + 캐싱)
>          → Claude API (일일 리포트)
> ```
>
> **DB 스키마 (핵심):**
> ```sql
> users (id, email, name, provider, created_at)
> portfolios (id, user_id, name, is_default)
> holdings (id, portfolio_id, symbol, category, avg_cost, shares, target_return, ...)
> transactions (id, holding_id, date, type, price, shares) -- 분할 매수/매도
> alerts_history (id, user_id, alert_type, symbol, message, created_at)
> ai_reports (id, user_id, symbol, report_date, content_json)
> ```
>
> **마이그레이션 전략:**
> - 로그인 전: localStorage 그대로 사용 (현재)
> - 로그인 후: localStorage → DB로 자동 마이그레이션
> - 비로그인 사용자도 계속 사용 가능 (localStorage)
